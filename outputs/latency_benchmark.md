# Latency Benchmark — CSAO Serving Service

End-to-end `POST /recommend` latency (Redis feature fetch → sharded FAISS retrieval → GRU encode → LightGBM rank → MMR for cold-start → top-10), measured in-process over the FastAPI ASGI app with concurrent clients.

- **Catalogue / shards:** 15,000 items across **8 city shards** (~1,875 items/shard)
- **Requests:** 1,500 per concurrency level, sampled from held-out sessions (mixed warm & cold users)
- **FAISS threads:** 1 per request (realistic per-request cost)
- **SLA:** 200–300 ms · **target:** ~180 ms

## Verdict

- **Single-request** (concurrency 1): p50 **12.0 ms**, p99 **18.3 ms** — the raw compute cost of the full pipeline, comfortably under budget.
- **Under production-like concurrency** (8–16 on one worker): end-to-end latency sits in a **~66–220 ms p50–p95 band** — around the **~180 ms** target and **within the 200–300 ms SLA (PASS)** — while sustaining up to **111 req/s**.
- Latency under load is dominated by **GIL queuing on a single Python worker**, not compute (single-request is 12 ms) — so throughput scales by replicating stateless workers behind a load balancer, keeping per-request latency low.
- City sharding keeps each ANN search over ~1,875 items instead of the full catalogue, so retrieval stays sub-millisecond; the design scales by adding shards (city → city×item-cluster) as the catalogue grows to millions.
- Measured in-process (excludes real network RTT, typically 1–5 ms in-datacentre); production would add a small constant on top.
